# MISSION.md 模板

> 来源：Matt Pocock `/teach` — MISSION-FORMAT.md

`MISSION.md` 位于学习主题根目录。每个教学决策 — 教什么、用什么资源、设计什么练习 — 都应该溯源到此文件。

## 模板

```md
# Mission: {主题}

## Why
{1-3 句话。用户追求的具体现实目标。学会后他们的生活或工作中会有什么变化？
避免抽象表述如 "理解XX" — 追问背后的真实结果。}

## Success looks like
- {一个具体的、可观察的能力 — "我能做XX"}
- {另一个具体的能力}
- {……}

## Constraints
- {时间/预算/既往承诺/学习偏好/任何限制方法的东西}

## Out of scope
- {用户明确不想碰的相邻主题 — 保护最近发展区}
```

## 规则

- **一个 workspace 一个 mission**。如果用户想学两个不相关的东西，需要两个独立workspace
- **具体优于抽象**。"10月前跑完半马" 优于 "变健康"。"给我的团队交付一个Rust CLI" 优于 "学会Rust"
- **模糊就追问**：如果用户说不清为什么学，在写之前先访谈他们。坏的 Mission 比没有 Mission 更糟
- **现实变了就修订**：Mission会变化。当用户目标变了，更新此文件 — 不要让过时的 mission 驾驭未来的session
- **保持简短**：如果 `MISSION.md` 跑超过一屏幕，它就不再是指南针而变成了计划书
